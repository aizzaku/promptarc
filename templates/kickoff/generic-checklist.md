# Generic Kickoff Checklist

Extends `universal-checklist.md`. Used when the project doesn't fit a specific domain.

---

## Clarification

1. **What's the primary objective?** (What does "done" look like for this specific task?)
2. **What format should the output be?** (Document | Spreadsheet | Presentation | Code | Conversation | Other)
3. **Who will consume this output?** (Yourself | Specific person/team | Public audience)
4. **What's the quality definition?** (Quick-and-dirty is fine | Needs to be presentable | Needs to be polished)
5. **What constraints exist?** (Time, format, length, tools, style, etc.)

## Context

6. **What have you tried so far?** (Prior attempts, what worked, what didn't)
7. **Is there a reference example?** (Something similar that represents the quality/style you want)
8. **What domain knowledge should I assume you have?** (Helps calibrate the depth and vocabulary of the output)

---

## That's It

The generic checklist is intentionally short. If a project needs more structured exploration, it probably belongs in a specific domain checklist. The `/arc-kickoff` skill can re-classify mid-interview if domain signals emerge from answers.
