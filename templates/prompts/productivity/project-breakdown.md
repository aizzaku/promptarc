# Pattern: Break Down an Overwhelming Project

> "I want to break down an overwhelming project into tasks"

**Principles used**: Chain-of-Thought Scaffolding, Specificity Amplification, Constraint-Driven Creativity

---

## When to Use
- A project feels too big to start
- You know the goal but not the steps
- Need to turn a vague objective into a concrete action plan

## The Pattern

```
I need to break down this project: {{PROJECT_DESCRIPTION}}

Goal: {{WHAT_DONE_LOOKS_LIKE}}
Deadline: {{DATE_OR_NONE}}
Time available: {{HOURS_PER_WEEK}}
Biggest blocker: {{WHAT_MAKES_THIS_FEEL_OVERWHELMING — scope? uncertainty? too many dependencies?}}

Break it into:
1. Phases: Group related work into 3-5 phases that deliver something tangible at each stage.
2. Tasks within each phase: Each task is a SINGLE ACTION a person can sit down and do in one session (30 min - 4 hours). "Research competitors" is not a task. "Read and take notes on Competitor X's pricing page" is a task.
3. Dependencies: Which tasks block other tasks? What can run in parallel?
4. The critical path: What's the sequence of tasks that determines the minimum project duration?

For task 1 of phase 1, make it the easiest possible action — something I can do in under 15 minutes. The first step's only job is to create momentum.

Include:
- Estimated hours per task (round up)
- Running total — does this fit in the time available before the deadline?
- If it doesn't fit: what to cut, defer, or simplify

Do NOT: Create tasks that are actually projects (multi-step, multi-session efforts masquerading as tasks).
```

## Why It Works
The 30-min-to-4-hour constraint on task size prevents tasks that are actually sub-projects. Starting with the easiest possible action reduces activation energy. The time budget check catches unrealistic plans early.

## Common Mistakes
- Tasks too large ("Build the frontend") → still feel overwhelming
- Tasks too small ("Open VS Code") → busywork that doesn't advance the project
- No dependency mapping → tasks get done in the wrong order
