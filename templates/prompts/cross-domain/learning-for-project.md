# Pattern: Create a Learning Plan for a New Tech Stack

> Domains: Learning & Research + Software Engineering

**Principles used**: Chain-of-Thought Scaffolding, Context Layering, Constraint-Driven Creativity

---

## When to Use
- Need to learn a new technology for an upcoming project
- Time-constrained learning (not academic — need to build with it)
- Want to get productive fast, not become an expert

## The Pattern

```
I need to learn {{TECHNOLOGY}} for a project: {{WHAT_I'M_BUILDING}}

My background: {{WHAT_I_KNOW_THAT'S_RELATED}}
Deadline: I need to be productive (not expert) in {{TIMEFRAME}}
Time available: {{HOURS_PER_WEEK}}

Build a project-oriented learning plan:

1. The 20% that covers 80%: What are the core concepts I must understand to be productive? (Not comprehensive — targeted to what I need for THIS project.)
2. Structured by building, not reading:
   - Mini-project 1 (day 1-2): {{SOMETHING_TRIVIAL_THAT_PROVES_THE_TOOLCHAIN_WORKS}}
   - Mini-project 2 (day 3-5): {{SOMETHING_THAT_EXERCISES_THE_CORE_CONCEPTS}}
   - Mini-project 3 (week 2): {{SOMETHING_CLOSE_TO_MY_ACTUAL_PROJECT}}
3. Reference material: For each concept, ONE resource (not five) — the single best explanation. Prefer official docs over tutorials.
4. Landmines: What are the non-obvious gotchas that waste beginners' time? Flag them early so I can avoid them.
5. When to stop learning and start building: What's the minimum knowledge threshold to begin the real project? (Hint: it's lower than you think.)

Do NOT: Design a comprehensive curriculum. I don't need to learn everything — I need to learn enough to build {{WHAT_I'M_BUILDING}}.
```
